RootCause AI

An AI tool that helps you figure out the root cause of a problem and explains it back to you through a chat, instead of just giving you a one-line answer.

by Yongye Uy · personal project

Live Demo → Frontend Repo Backend Repo

what it doesOverview

When something breaks, it's easy to fix the symptom without ever finding out why it actually happened. RootCause AI takes a problem you describe, uses AI to dig into what's really causing it, and lets you keep chatting with it to ask follow-up questions until it makes sense. This is my most recent project and the one I'm most proud of, since it involved building and deploying a real full-stack app, not just a script.

what's builtKey Features

  • AI-powered root cause analysis — you describe a problem and the AI breaks down what's likely causing it.
  • A chat-style interface so you can keep asking questions and get more explanation.
  • A backend API that talks to the AI service and handles requests from the frontend.
  • A health check endpoint so I can tell if the backend is actually running.
  • Login system (depending on the current version being worked on).
  • Deployed to the cloud so anyone can try it, not just run it locally.

not as easy as it soundsInteresting Challenges

Getting this project actually running online taught me a lot about real-world deployment, not just writing code that works on my own computer.

  • The free server kept falling asleep. Hosting the backend on a free tier (Render) meant it would go to sleep after being idle, making the first request after a while very slow.
  • Keeping it awake with a cron job. I set up a scheduled job that pings the backend regularly so it doesn't fall asleep on real users.
  • Moving the backend. I had to migrate the backend at one point, which meant carefully checking that nothing broke along the way.
  • Choosing a database. I was weighing Neon PostgreSQL against a simple SQLite file, thinking about what would actually work once the app was deployed rather than running locally.
  • Serverless architecture. Learning how a backend behaves when it's not just "always running" like on my own machine.
  • Getting Vercel deployment right. Making sure the frontend build and environment settings actually worked in production, not just in local testing.
